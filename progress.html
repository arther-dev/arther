How status is derived (and its limits)

The task list (the blocks) comes from the four canonical phase task docs (arther-phase{1..4}-tasks.md) — the plan itself names these as the source of the ~213 tasks. Done/blocked/not-started is inferred from the §10 Session log in IMPLEMENTATION_PLAN.md, which records merged work:

  • Shipped — the task ID appears in the log in a non-deferred context, or its epic/milestone is declared complete (e.g. “F5/F6 closed”, “M5 — Generate — is structurally complete”). Milestone→epic mapping is read live from §4.
  • Blocked / gated — the ID appears only alongside blocking language (e.g. “blocked on a paid plan”, “gated on Trigger.dev creds”, “needs Feature 4”). Currently: .
  • Not started — never referenced as shipped, and its epic isn’t complete.

This is a best-effort read of prose, not a formal status field, so a few foundational tasks logged without their ID may under-report. Click a block to set a local override (stored in your browser); reset overrides.
